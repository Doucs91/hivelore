# Benchmark v0.2.7 - Measured Token-Reduction Results

Test run on 2026-04-28 on sandaga-monorepo (Spring Boot + React, ~1.3GB).
Methodology: 4 agents in parallel, 2 identical tasks with/without hAIve.

## Aggregate Results (2 Tasks)

| Metric | Without hAIve | With hAIve | Reduction |
|---|---|---|---|
| Total tokens | 73,487 | 49,540 | **-32.6%** |
| Tool calls | 57 | 17 | **-70.2%** |
| Total duration | 2 min 45 s | 1 min 44 s | **-36.4%** |
| Files read | 23 | 6 | **-73.9%** |

## By Task Type

**Complex task (payment architecture)**:
- Without hAIve: 59,278 tokens, 47 calls, 20 files read
- With hAIve: 36,799 tokens, 15 calls, 6 files read
- Gain: **-38% tokens, -68% calls**

**Targeted debugging task (Flyway checksum)**:
- Without hAIve: 14,209 tokens, 10 calls, 3 files read
- With hAIve: 12,741 tokens, 2 calls, 0 files read
- Gain: **-10% tokens, -80% calls, -32% time**

## Key Observations

1. **The more architectural the task, the higher the gain** - exploratory tasks benefit more than targeted debugging tasks.
2. **The bridge file (CLAUDE.md) provides passive value** - even without explicitly calling hAIve, the non-hAIve agent benefited from the `CLAUDE.md` generated by `haive init`.
3. **hAIve is only as good as what it is given** - the non-hAIve agent discovered a gotcha (`mobile_payments` without migration) that was missing from memories; after the benchmark, that memory was added.
4. **-70% tool calls = direct latency reduction** - each avoided call saves roughly 1-2s of latency.

## Conclusion

The main gain is not only saved tokens but **reduced trial-and-error**: the agent with hAIve reaches the right file, with the right class name, on the first attempt.

## How to apply

Use these benchmark numbers as directional ROI evidence, not as a universal guarantee. When presenting hAIve externally, prefer reproducible current benchmarks from `haive bench` or `haive benchmark report`, and cite this record only as historical context for why reducing exploration/tool calls matters.
