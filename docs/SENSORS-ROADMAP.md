## Status of the sensors feature (what works today)
- `SensorSchema` on memory frontmatter (`packages/core/src/schema.ts`); `kind: regex` executed,
  `shell`/`test` recognized but not run.
- Engine `packages/core/src/sensors.ts`: per-file diff parsing (`sensorTargetsFromDiff`), strict path
  scoping (`normalizeProjectPath` → exact file or `dir/` prefix), `runSensors`/`runRegexSensor`.
- Assisted generation `packages/core/src/sensor-suggest.ts` wired into `mem_save`, `mem_tried`,
  `haive memory add`, `haive memory tried`. Autogen sensors are always `severity: warn`, `autogen: true`.
- `anti_patterns_check` surfaces a `"sensor"` reason (ranked highest); `precommit-check.ts` maps a
  `severity: block` sensor hit to a deterministic block.
- CLI `haive sensors list | check | export` (`packages/cli/src/commands/sensors.ts`). `check` exits 1
  only on a `block`-severity hit; `export` emits a runnable grep script (and an eslint JSON stub).

## Planned follow-ups

### A. `haive sensors promote <id>` — flip an autogen sensor to `block`
**Why:** autogen sensors are intentionally `warn`-only, so the deterministic *blocking* value only
turns on after a human vets them. Today that means hand-editing frontmatter. A one-liner closes the
loop: review the warn-firing sensor, then promote it to an enforced guardrail.
**How:**
- New subcommand in `packages/cli/src/commands/sensors.ts`: `haive sensors promote <memory-id>
  [--severity block|warn] [--dir]`. Load the memory, set `sensor.severity`, re-serialize
  (`serializeMemory` strips `undefined`, so keep the block well-formed), write back.
- Guardrails: only operate on memories that already have a `sensor`; print the pattern + message and
  require `--yes` (or confirm) before promoting to `block` (promotion makes the gate hard-fail).
- Optional companion: `haive sensors demote <id>` / `--severity warn` to soften.
- Tests: CLI integration test (promote sets `severity: block`; non-existent/sensorless id errors cleanly).

### B. Sharper autogenerated patterns (capture the offending *value*, not just the key)
**Observed:** for "open-in-view=true is bad", `suggestSensorFromMemory` extracted `pattern: open-in-view`,
which also matches the *correct* line `open-in-view=false`. The MCP/CLI `check` only scans **added**
diff lines so it's fine in the gate, but the **exported grep script** matches every occurrence
(including `=false`), producing noisy hits.
**How:**
- In `packages/core/src/sensor-suggest.ts`, prefer a key+value span when the body shows an assignment
  (e.g. `` `open-in-view=true` ``, `status = "KO"`, `rec_7`): build the pattern from the full
  bad token incl. its value/operator, not just the identifier. Keep the conservative "return null when
  unsure" stance.
- Add `sensor-suggest.test.ts` cases: assignment-style body → pattern includes the value; plain
  identifier body → unchanged behavior.

## Minor review observations (not bugs — design notes)
1. **Autogen sensors never block until promoted.** Correct, safe default; pairs with follow-up A.
   Document this clearly in user-facing help so teams know a warn sensor is *advisory* until promoted.
2. **Broad pattern vs exported grep (see B).** Gate is safe (added-lines only); the standalone grep
   export is the noisy surface. Fixing B resolves this; alternatively the grep emitter could scope to
   added lines only, but that loses the "scan the whole repo" value — prefer sharpening the pattern.
3. **`memory tried` flag shape is `--what/--why-failed/--instead`, not `--body`.** Well-documented in
   `--help`; noting it because it's an easy first-time trip for agents. No change needed unless we want
   to accept `--body` as an alias.
4. **`haive` resets the shell cwd between invocations in some environments.** Operationally, pass
   `--dir <root>` to `sensors list/check/export` rather than relying on `cd`. Not a code issue; worth a
   line in docs/examples.
5. **Exported eslint format is a JSON stub**, not real ESLint rules yet. If teams want true ESLint
   integration, emit a flat-config snippet or a custom-rule plugin in a later phase.
